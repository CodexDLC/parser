# Project Brief

## Идея

Project M4 - учебный backend-сервис для автоматизации новостного Telegram-канала. Сервис собирает новости из разных источников, отбирает релевантные материалы, превращает их в короткие посты через AI и публикует в Telegram.

## Пользовательская цель

Администратор канала должен иметь возможность:

- добавить сайт или Telegram-канал как источник новостей;
- задать ключевые слова и фильтры;
- запустить ручную генерацию поста;
- видеть историю новостей, постов и ошибок;
- включить автоматическую публикацию по расписанию.

## MVP

Минимально полезная версия проекта должна включать:

1. FastAPI-приложение со Swagger-документацией.
2. CRUD для источников новостей.
3. CRUD для ключевых слов.
4. Хранение новостей и постов в базе данных.
5. Один рабочий сайт-парсер.
6. Один рабочий Telegram-парсер через Telethon.
7. Фильтрацию по ключевым словам и дедупликацию.
8. Ручную AI-генерацию через API.
9. Celery-задачи для парсинга, генерации и публикации.
10. Публикацию в Telegram-канал через Telethon.
11. README с инструкцией запуска.

## Не входит в первый MVP

Эти вещи можно добавить позже, если останется время:

- полноценная веб-панель администратора;
- сложная система ролей и авторизации;
- ML-классификация тем;
- сложная аналитика публикаций;
- поддержка десятков разных сайтов;
- автотесты для реальных внешних API без моков.

## Главные технические решения

- Backend: FastAPI.
- Очередь задач: Celery.
- Broker: Redis.
- База данных: PostgreSQL как основная и обязательная БД проекта.
- ORM: SQLAlchemy.
- Миграции: Alembic, если проект успевает выйти за пределы простого MVP.
- AI: OpenAI-compatible client, чтобы не привязывать доменную логику к одному провайдеру.
- Telegram: Telethon.

Дополнительный production-вариант не отменяет учебный контракт: чтение Telegram и
демонстрационная публикация остаются на Telethon, а для RSS → AI → собственный канал
можно явно выбрать отдельный Bot API publisher.

## Стиль структуры кода

Проект строим не в стиле нескольких больших файлов, а в более модульной структуре:

- каждая ORM-модель живет в своем Python-файле;
- каждая группа Pydantic-схем живет в своем файле;
- каждый repository/service/parser/client выносится в отдельный файл, если это отдельный класс или самостоятельная ответственность;
- `__init__.py` используется только для удобного экспорта, а не как место для большой логики.

Такой стиль немного увеличивает количество файлов, зато проект легче читать, расширять и сдавать по частям.

## Критерии успеха

Проект считается готовым к сдаче, если:

- можно поднять API локально;
- Swagger открывается по `/docs`;
- можно добавить источник и ключевое слово через API;
- можно получить новости хотя бы из одного сайта и одного Telegram-канала;
- можно сгенерировать пост вручную через API;
- Celery-задачи запускаются и пишут результат в БД;
- публикация в Telegram работает или имеет безопасный dry-run режим;
- README объясняет запуск и демонстрацию.
